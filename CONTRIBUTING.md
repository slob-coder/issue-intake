# Contributing — How to Submit a Request

Thank you for your interest in using OpenClaw! This guide explains how to submit a development request and what to expect.

## Submitting a Request

### Step 1: Create an Issue

1. Go to the [Issues tab](../../issues)
2. Click **New Issue**
3. Select the **🚀 Feature Request** template
4. Fill out all required fields

### Step 2: Write a Good Description

The better your description, the better the result. Include:

- **What** you want built (be specific)
- **Why** you need it (what problem it solves)
- **How** you envision it working (examples help!)

**Good example:**
> A CLI tool that converts Markdown files to PDF. It should support GitHub Flavored Markdown, syntax highlighting for code blocks, and custom CSS themes. Output should be a single PDF file.

**Vague example (avoid):**
> Make me a markdown tool

### Step 3: Specify Requirements

List the specific features and behaviors you need:

- Use bullet points for clarity
- Prioritize what's essential vs nice-to-have
- Include any technical preferences (language, framework, platform)

### Step 4: Wait for Design Review

Our AI designer will analyze your request and post a technical design document as a comment on your issue. This usually happens within 30 minutes.

### Step 5: Review and Approve

When the design is posted:

- **To approve:** Reply with 👍, `LGTM`, `approved`, or `OK`
- **To request changes:** Reply with your specific feedback

The system will revise the design based on your feedback and post an updated version.

> ⚠️ **Important:** You have **72 hours** to respond. After that, the issue will be automatically closed as stale. You can always reopen it or create a new one.

### Step 6: Track Progress

Watch the issue labels for status updates:

1. `designing` → AI is designing your project
2. `awaiting-feedback` → Your review is needed
3. `in-progress` → Development started
4. `in-review` → Code review in progress
5. `testing` → Automated testing
6. `releasing` → Preparing release
7. `completed` → Done!

### Step 7: Get Your Project

When complete, a comment will be posted with:
- A link to the project repository
- A link to the GitHub Release
- Download/installation instructions

## Guidelines

### What We Can Build
- CLI tools and scripts
- Web applications
- Libraries and packages
- Data processing tools
- Automation workflows

### What's Out of Scope
- Projects requiring paid APIs or services
- Projects involving sensitive personal data
- Very large projects (multiple repos, complex infrastructure)
- Anything violating GitHub ToS or applicable laws

### Tips for Better Results
1. **Be specific** — Clear requirements lead to better outcomes
2. **Include examples** — Links to similar tools or screenshots help a lot
3. **Specify constraints** — If you need a specific language or platform, say so
4. **Keep it reasonable** — Start with a focused MVP; you can always request more later
5. **Respond to design reviews** — Your feedback improves the final product

## Questions?

If you have questions about the process or need help with your request, feel free to open a discussion or add a comment to your issue.
